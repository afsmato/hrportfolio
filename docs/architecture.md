## エラーレスポンス標準

すべての API Routes は以下の形式でエラーを返す。

| HTTPステータス | 意味 | 使用場面 |
|---|---|---|
| 400 | Bad Request | Zodバリデーション失敗・不正な入力値 |
| 401 | Unauthorized | セッションなし・セッション期限切れ |
| 403 | Forbidden | 他ユーザーのリソースへのアクセス試行 |
| 404 | Not Found | 指定IDのリソースが存在しない |
| 500 | Internal Server Error | 予期しない例外（外部API障害含む） |

**レスポンスボディ形式**:
```json
{ "error": { "code": "VALIDATION_ERROR", "message": "skillId is required" } }
```

**エラーコード一覧**:

| code | 対応するエラークラス | 説明 |
|---|---|---|
| `VALIDATION_ERROR` | `ValidationError` | 入力値がバリデーションルール違反 |
| `NOT_FOUND` | `NotFoundError` | リソースが存在しない |
| `EXTERNAL_API_ERROR` | `ExternalApiError` | 外部API（楽天・Amazon・Claude）の呼び出し失敗 |
| `UNAUTHORIZED` | - | 認証エラー |
| `FORBIDDEN` | - | 認可エラー |
| `INTERNAL_ERROR` | - | 予期しない例外 |

## データ永続化戦略

### ストレージ方式

| データ種別 | ストレージ | 理由 |
|-----------|----------|------|
| ユーザー・スキル評価・学習ログ | PostgreSQL (Neon) | リレーショナルデータ、JOINが必要 |
| 収集済み記事 | PostgreSQL | 全文検索・タグ検索が必要 |
| 書籍データ | PostgreSQL | ISBNを共通キーにした正規化が必要 |
| トレンドレポート | PostgreSQL | TrendReport・TrendKeywordのリレーション管理 |
| 通知ログ（送信済み・バナー表示フラグ） | PostgreSQL | ユーザーごとのリマインダー状態・アンケート回答状態を管理 |
| セッション情報 | NextAuth.js（DBセッション） | セキュリティ上DBで管理 |
| 外部APIキー・キャッシュ | Vercel環境変数 / Upstash Redis | クライアントに露出させない / 楽天・Amazon APIのレート制限対策 |

### バックグラウンド処理（Vercel Cron Jobs）

| ジョブ | 頻度 | 処理内容 |
|---|---|---|
| 記事収集 | 毎日 06:00 JST | RSSフィードから新着記事取得 → Claude APIで要約・タグ付け → DB保存 |
| 書店ランキング更新 | 毎週月曜 07:00 JST | 楽天 + Amazon からビジネス書ランキング取得 → HR関連フィルタ → DB更新 |
| トレンドキーワード生成 | 毎月1日 08:00 JST | 直近2ヶ月の記事データをClaude APIで分析 → TrendReport生成・保存（約3,000〜5,000トークン/回） |
| スキル棚卸しリマインダー | 毎日 09:00 JST | 最終スキル評価から90日超のアクティブユーザーを抽出 → Resendでメール送信 → 通知ログ保存 |

#### Cron Job 設計上の注意事項

- **タイムアウト制限**: Vercel Proプランの最大実行時間は60秒。記事収集ジョブはClaude APIを50件分連続呼び出しするため、**バッチ分割（例: 10件ずつ）** して部分成功を許容する設計にする
- **エラーハンドリング方針**: 個別記事の要約失敗は警告ログのみでスキップし、ジョブ全体を止めない。メール送信失敗は翌日のジョブで再送信する（冪等性を担保）
- **Claude API コスト試算**:
  - 記事収集（日次）: 約500トークン/記事 × 50記事 = 25,000トークン/日
  - トレンド生成（月次）: 約5,000トークン/回
  - 月間合計: 約755,000トークン（月間上限アラートの設定基準値として使用）

### バックアップ戦略

- **方式**: Neonの自動バックアップ（日次）
- **保持期間**: 7日間
- **復元方法**: Vercelダッシュボードからポイントインタイムリストア

## パフォーマンス要件

### レスポンスタイム

| 操作 | 目標時間 | 測定環境 |
|------|---------|---------|
| ダッシュボード初回表示 | 2秒以内 | スマートフォン・4G回線 |
| スキルマップ更新（自己評価） | 500ms以内 | 同上 |
| 記事フィード表示 | 1秒以内 | 同上 |
| 書籍検索（楽天 + Amazon） | 3秒以内 | 同上（外部API依存） |

### 最適化戦略

- **記事フィード**: ISR（インクリメンタル静的再生成）で1時間ごとに更新
- **スキルマップ**: クライアントサイドのSWRでUIをオプティミスティック更新
- **外部API**: Upstash Redisでレスポンスをキャッシュ（楽天・Amazon APIのレート制限対策）
  - キャッシュキー: `books:search:{クエリハッシュ}` / `books:ranking:{日付}`
  - TTL: 書籍検索結果は24時間、週次ランキングは7日間
  - キャッシュミス時のみ外部APIを呼び出し、結果をKVに保存してからレスポンス

## セキュリティアーキテクチャ

### 認証・認可

```
ユーザー → NextAuth.js（Google/GitHub OAuth）
         → JWTセッション（HTTP-Only Cookie）
         → API Routes（セッション検証）
         → サービスレイヤー（userId で操作を制限）
```

### データ保護

| 対象 | 対策 |
|---|---|
| ユーザーデータ | 自分のデータのみアクセス可（userId フィルタ） |
| 外部APIキー | Vercel環境変数（サーバーサイドのみ） |
| セッション | HTTP-Only Cookie、CSRF対策はNextAuth.jsが担保 |

### 入力検証

- すべてのAPI Routesで **Zod** によるスキーマバリデーション
- SQLインジェクション対策は **Prisma** のパラメータ化クエリで担保
- 外部RSS・API取得データはサニタイズしてDB保存
  - RSS記事: タイトル・要約のHTMLタグをDOMPurify相当の処理で除去、URLはhttps://ドメインのみ許可
  - 書籍API: 書籍名・著者名の文字数上限チェック（各256文字以内）、imageUrlはURLバリデーションのみ

## 移行容易性設計（Vercel → Google Cloud）

現フェーズはVercelで構築するが、ユーザー数が5,000人を超えた段階でGoogle Cloudへの移行を検討する。
移行コストを最小化するため、以下の方針でVercel依存を意図的に抑制する。

### Vercel依存を避ける設計方針

| Vercel固有サービス | 方針 | 代替先（移行時） |
|---|---|---|
| Neon（PostgreSQL） | **使用する**（標準PostgreSQL互換のため移行容易） | Cloud SQL for PostgreSQL |
| Upstash Redis（Redis互換） | **使用する**（Redis互換のため移行容易） | Cloud Memorystore (Redis) |
| Vercel Cron Jobs | **使用する**（移行時はコード変更なし・設定のみ変更） | Cloud Scheduler + Cloud Run Jobs |
| Vercel Blob（独自ストレージ） | **使用しない** → 代わりにS3互換APIを使う | Cloud Storage |
| Edge Functions（Vercel独自） | **使用しない** → 標準API Routesのみ使う | Cloud Run |

### アプリケーションの移植性確保

```
【守る設計原則】
✅ Next.jsアプリはDockerコンテナでも動く構成にする
✅ DBアクセスはPrismaのみ経由（生SQLを書かない）
✅ 環境変数で接続先を切り替えられる設計
✅ Vercel固有のAPIやSDKをアプリコードに直接書かない

【移行時の作業イメージ】
Neon          → Cloud SQL:    接続文字列を環境変数で切り替えるだけ
Upstash Redis      → Memorystore:   同上
Vercel Cron    → Cloud Scheduler: cronの設定をVercel→GCPに移すだけ
アプリ本体     → Cloud Run:     Dockerfileを追加してデプロイするだけ
```

### 移行判断の目安

| 指標 | 移行検討開始 | 移行実施 |
|---|---|---|
| 月間アクティブユーザー | 3,000人超 | 5,000人超 |
| 月額インフラコスト | $500超 | $1,000超 |
| Vercel制約への抵触 | 関数実行時間・転送量の上限に近づく | 上限に達する |

### データ増加への対応

| データ | 1年後の想定量 | 対策 |
|---|---|---|
| 記事 | 約18,000件（50件/日） | `publishedAt` にインデックス、古い記事はアーカイブフラグ |
| 学習ログ | ユーザーあたり約1,200件/年 | ページネーション、月別集計テーブルのキャッシュ |
| スキル評価履歴 | ユーザーあたり約72件/年（全スキル×4回） | 最新評価のみ表示、履歴は別テーブル |

### 領域拡張への対応（P2: 他HR専門領域の追加）

スキルフレームワークはコード定数（`SKILL_FRAMEWORK`）で管理しているため、新領域の追加はコード変更のみで対応可能。DBマイグレーションは不要。

### ドメイン定数（`src/constants/DOMAIN_CONSTANTS.ts`）

ビジネスルールに関わる数値定数は一元管理し、複数サービス間の不整合を防ぐ。

```typescript
export const DOMAIN_CONSTANTS = {
  // ギャップスキル判定: スコアがこの値以下をギャップとみなす
  GAP_THRESHOLD: 2,

  // 棚卸しリマインダー: 最終評価からこの日数経過でリマインドを送信
  REVIEW_INTERVAL_DAYS: 90,

  // 成長実感アンケート: 初回スキルマップ完成日からこの日数後に表示
  // （フェーズ2ではサマリ生成ごとに実施するため、フェーズ1のみ適用）
  SURVEY_TRIGGER_DAYS: 90,
} as const;
```

## テスト戦略

### ユニットテスト（Vitest）

- **対象**: サービスレイヤーのビジネスロジック
  - `calcArticleScore`（おすすめ記事スコアリング）
  - `SkillService.getGapSkills`（ギャップスキル算出）
  - `PortfolioService.generatePortfolio`（データ集約）
- **カバレッジ目標**: サービスレイヤー 80%以上

### 統合テスト（Vitest + テスト用DB）

- **対象**: API Routes → サービスレイヤー → DBの連携
  - スキル評価 → スキルマップ更新 → おすすめ記事変化
  - 書籍検索 → 積読追加 → 読了更新 → ポートフォリオ反映
- **方式**: テスト専用のPostgreSQLコンテナを使用（Docker Compose）
  - devcontainer環境ではDockerが利用可能なためpg-memより信頼性が高い
- **完了基準**: P0・P1の全APIエンドポイントについてハッピーパス + 主要エラーケースを網羅

### E2Eテスト（Playwright）

- **シナリオ**:
  - 新規ユーザーがスキルマップを完成させるまでの初回フロー
  - 記事を積読に追加 → 読了メモを記録するフロー
  - 書籍アンケートに回答 → 学習キューに追加されるフロー
- **完了基準**: 全シナリオがCIでマージ前にパスすること
- **実行環境**: モバイルエミュレーション（iPhone 14相当）を含む

## 技術的制約

### 環境要件

- **対応ブラウザ**: Chrome / Safari / Firefox 最新2バージョン
- **対応デバイス**: スマートフォン（iOS 16+ / Android 12+）、PC
- **外部依存**:
  - 楽天ブックスAPIアカウント
  - Amazon アソシエイトアカウント（PA API利用に必要）
  - Claude API キー
  - Vercel アカウント（Postgres + Cron Jobs + KV）
  - Resend APIキー（メール送信）

### パフォーマンス制約

- 楽天ブックスAPI / Amazon PA APIのレート制限に準拠（書籍検索はUpstash Redisでキャッシュ）
- Claude APIの課金制約：月間約755,000トークン想定（内訳はCron Job設計上の注意事項を参照）。月間上限アラートを設定して管理

### セキュリティ制約

- Amazon PA APIのリンク表示ルールに準拠（アフィリエイトリンクとしての表示義務）
